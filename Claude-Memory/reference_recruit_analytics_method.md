# 採用データ分析メソッド（会社非依存の型）

ぴゅあ訪問看護グループ(株式会社ジョン)の分析で確立した手順を、他社にも回せる「型」に一般化。
具体事例は [[report-jobmedley-recruit-20260616]]（スカウト）/ [[report-recruit-seo-panel-20260616]]（SEO）/ [[project-recruit-data-team]]（自動点検）。既存の集計型は [[reference-recruit-analysis-generic-playbook]]。

## 大原則（7つ）
1. **媒体を混ぜない**：1媒体=1ファネルで測る（混ぜると何が効いたか不明になる）。
2. **計測の健全性を最初に疑う**：そもそも正しく測れているか（データが本物か）を一番先に確認。
3. **率で見る**：数ではなく通過率・前段比。
4. **反応セグメントは"診断"であってKPIではない**：誰が反応するか≠誰を採れるか。
5. **崩落は下流に多い**：量(集客)より配管(訴求・フォロー・クロージング)。
6. **専門家パネルで単眼バイアスを潰す**：重要判断は複数視点＋統計家で。
7. **事実は捏造しない・条件は断定しない**：未確認は[ ]プレースホルダ→承認後に確定。

## STEP1｜データ源の棚卸し
チャネル別に「何があるか」を並べる：
- 認知/集客：SNS各媒体・求人媒体・検索(SEO)
- サイト：GA4＋Search Console
- 中間：LINE/メッセージツール(Lステップ等)
- 応募経路：媒体スカウト／媒体直応募／直接(電話・HP・リファラル)
- 選考：応募者管理(書類→面接→内定→入職)
取得手段：①エクスポートCSV ②管理画面コピペ(Chrome) ③API自動転記(GAS)。

## STEP2｜計測健全性チェック（最重要・最初にやる）
「測れているつもり」を疑う。実際にあった故障例：
- **イベント名の不一致**：計測が待つ合言葉とサイトが送る合言葉が違う→CV永久0。
- **空の箱を追っている**：使われていないプロパティ/コンバージョン設定。
- **一覧が出さない値を一律"なし"で取得**：スカウトの「返信」が全行"なし"＝取りこぼし（実際は返信あり）。
- **大きいシートの途中truncation**：「投稿停止」と誤検知。
点検：受信しているか／ホスト名／イベント発火／**全行が同一値＝異常を疑う**／最終更新日(鮮度)。

## STEP3｜ファネル定義（媒体ごと・混ぜない）＋ゴールを1つ
- SNS：配信(reach)→視聴→**行動転換(プロフ遷移/フォロー/保存)**
- SEO：**表示(imp)→クリック(CTR)→着地**（順位は主KPIにしない）
- スカウト：**送信→開封→返信→書類選考→面接設定**
- LINE：登録→純増→エントリー
各媒体のゴールを明示（例：スカウト=面接設定）。**直応募・入職は別管理**（入職は媒体横断で）。

## STEP4｜率で読む
- 数でなく通過率・前段比で見る。
- **SEOの掲載順位は信用しない**（全表示の平均＝合成値・地元IPバイアス）→ imp/CTRで判断。
- **小さい分母の率は捨てる**（imp10未満、N小の「1位/100%」はゴミ）。

## STEP5｜反応セグメント分析（誰が反応するか）
- 反応者(warm＝手挙げ/返信)の属性 vs 送付対象(cold)の属性を比較。
- **★最大の落とし穴**：差分は「反応者の魅力」ではなく〈①撃っていない象限の地図＋②市場で動きにくい層の選択効果〉の二重写し。**採否予測に使わず"診断・撃ち先補正"にのみ使う。**
- cold側の不自然な絞り（例：年齢で機械的に外している）を点検。

## STEP6｜崩落点の特定
- どの段で落ちるかを率で見る。多くは**下流**（訴求の不安後出し／フォロー無し／クロージング遅い）。
- 「断られている(辞退)」か「無視(未開封)」かで打ち手が違う。
- **辞退/脱落の"理由"を必ずコード化**：条件・距離・時期・(オンコール等の固有不安)・他社決定。→ 追加計測ゼロで実因が出る最大の情報源。

## STEP7｜専門家パネル（重要判断時）
- 複数の専門視点で議論→収束。**統計家を必ず入れて**小N・代理指標・生存者バイアスの罠を潰す。
- 単眼の思い込み（例「反応多い層＝医療事務を狙え」）を相互批判で却下できる。
- 実装：5〜6人×2〜3ラウンド（独立分析→相互批判→収束→統合）。

## STEP8｜測定設計
- **記録**：1行=1アクション（会員番号/属性/施策ラベル/反応/進捗）。会員番号を突合キーに。
- **集計**：セグメント×ファネル＋率（ペルソナ/リスト種別/属性/フォロー有無）。
- **週次KPIは3つに絞る**（媒体のゴールに直結するもの）。**反応者の属性はKPIにしない**。

## STEP9｜改善実装
- **ペルソナ別文面**（属性は手入れで記録しA/B比較）。
- **フォロー標準化**（送りっぱなし厳禁＝3タッチ Day0/3/7）。
- **SLA**（応募→即日連絡／面接N日以内／内定→書面）。
- 不安要因（業界固有：訪問看護ならオンコール）は**入口で先出し**して後出し辞退を消す。
- 未確認の数値・条件は[ ]プレースホルダ→責任者承認後に確定稿（YELLOW運用）。

## データ操作Tips（実務）
- 大CSV/シート：download xlsx + openpyxl(data_only=True) か、UTF-8レポートをファイル出力（Windowsコンソール文字化け回避）。read_file_content はtruncateするので保存ファイルから抽出。
- Chrome Claude：管理画面を**パイプ区切り表**で抽出させ、会員番号をキーに突合。ペルソナ・リスト種別は手入れ列。
- GAS：GA4＋Search Consoleをシートに毎日自動転記（Search ConsoleはAdvanced Serviceに無いのでUrlFetchApp+getOAuthToken、専用GCPプロジェクトに紐付け）。
- 個人情報：会員番号と属性のみ・氏名/連絡先は出さない。
